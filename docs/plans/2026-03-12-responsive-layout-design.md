# BizClaw Responsive Layout Design

**Date:** 2026-03-12

**Problem**

当前 BizClaw 桌面界面主要依赖两档断点：
- 默认桌面布局
- `1180px` 以下单列
- `760px` 以下进一步堆叠

这导致两个核心问题：
- 在 `1280x800 ~ 1440x900` 的小屏笔记本上，界面仍然保留接近大屏的留白、卡片尺寸和日志高度，垂直空间被过度占用。
- 在 `1920x1080` 的常见桌面屏上，整体比例略松散，顶部、主工作区和底部状态栏之间的层级不够均衡。

**Goal**

让界面在 `1280x800 ~ 1440x900` 和 `1920x1080` 两类主流分辨率下都保持：
- 可完整浏览核心操作区
- 信息层级清晰
- 控件尺寸舒适
- 底部状态栏不抢占主内容空间

**Non-Goals**

- 不调整业务逻辑、数据流和功能顺序
- 不引入新的页面结构或路由
- 不做移动端优先重构
- 不引入新的 UI 组件库

## Current Structure

主界面由四段组成：
- 顶部栏 `topbar`
- 流程条 `workflow-strip`
- 主工作区 `workspace-grid`
- 底部状态栏 `environment-bar`

当前响应式问题主要集中在三个层面：
- **全局节奏**：`dashboard-shell`、`surface-card`、标题字号、按钮和输入框尺寸在中等屏幕下过大。
- **主工作区比例**：`workspace-grid` 在小屏笔记本上仍偏向宽双栏，右侧运行区和左侧表单区互相挤压。
- **垂直空间管理**：日志卡片、支持信息卡、按钮排布和状态栏会累积占用较多高度。

## Approaches Considered

### Approach 1: Add a desktop-responsive scale system

在现有样式上新增一套桌面级断点和尺寸变量，统一收敛间距、字号、卡片内边距、按钮高度、日志高度和状态栏尺寸。

**Pros**
- 改动集中在 `src/styles.css`
- 风险小，不动业务逻辑
- 最适合当前单页桌面工作台结构

**Cons**
- 需要梳理较多尺寸常量
- 需要补几条界面测试来锁住关键布局

### Approach 2: Component-level adaptive layout

对每一块卡片单独设计自适应策略，甚至使用更细颗粒的容器感知模式。

**Pros**
- 精度更高
- 每块区域能更独立地优化

**Cons**
- 改动面更大
- 回归成本更高
- 当前页面结构下属于过度设计

### Approach 3: Only shrink type and padding

仅缩小字号、padding 和按钮高度。

**Pros**
- 工作量最小

**Cons**
- 不能从根本上解决 `1280` 宽附近的布局拥挤
- 右侧运行区和底部状态栏的比例问题仍会存在

## Chosen Direction

采用 **Approach 1**。

核心原则：
- 保持现有视觉语言不变
- 通过断点和变量让界面在不同分辨率下“成比例地收缩或舒展”
- 优先保证 `1280x800 ~ 1440x900` 的可用性，再兼顾 `1920x1080` 的均衡观感

## Breakpoint Strategy

### Base: large desktop

适用于大于 `1440px` 的桌面宽度。

保持当前结构，但会把一些硬编码尺寸改成变量，便于后续分档。

### `<= 1440px`

目标：让 `1440x900` 看起来更紧致，但不丢大屏层级。

调整方向：
- 缩小 `dashboard-shell` 外层留白
- 略微缩小 `workspace-card` 和 `topbar` padding
- 缩小 `workflow-step`、`support-tile`、`console-card` 的内边距
- 轻微收紧标题和副文案尺寸

### `<= 1280px`

目标：让 `1280x800` 下主操作区更容易完整出现在首屏。

调整方向：
- 收紧全局垂直节奏：按钮、输入框、错误条、日志区、状态栏都进一步压缩
- `workspace-grid` 改为更偏小屏的双栏比例，右栏不再占用过宽空间
- 支持信息卡和运行状态卡更紧凑
- 底部状态栏项目允许更合理的换行和最小宽度

### `<= 1180px`

目标：平滑转入单列。

调整方向：
- 保留当前单列思路
- 但补充对状态栏、按钮区、表单区和日志区的专门优化，避免只是简单堆叠

### `<= 760px`

保持当前移动/窄屏策略，并继承上面的变量收敛。

## Layout Changes

### 1. Global sizing tokens

把以下尺寸改为变量或 `clamp()`：
- 页面外边距
- 卡片内边距
- 标题字号
- 输入框和按钮高度
- 日志区最小高度
- 状态栏项目最小宽度和高度

这样不同断点只需要改变量，不必在多个类中重复手写数值。

### 2. Workspace grid behavior

`workspace-grid` 将做三段式布局：
- 大屏：保持偏宽双栏
- 中屏：收紧为更平衡的双栏
- 较窄桌面：切单列

目标是让左侧配置区在小屏笔记本上优先完整可用，右侧运行区仍保留可见性。

### 3. Vertical density tuning

重点压缩这些区域：
- `support-grid`
- `form-grid`
- `toggle-row`
- `button-row`
- `console-card--logs`
- `environment-bar`

其中日志区会采用更小的最小高度和更早的内部滚动，避免整页被日志拖长。

### 4. Status bar behavior

底部状态栏继续保持 sticky，但在中小桌面屏幕上：
- 降低外层 padding
- 缩短状态 pill 最小宽度
- 保持文字可读，不再抢占主工作区高度

## Testing Strategy

测试分两层：

### Regression tests

在 `src/App.test.ts` 增加或扩展断言，锁定：
- 底部状态栏语义和项目数量
- 状态栏 detail 的单行省略策略
- 与响应式类名相关的关键结构不被改坏

### Manual viewport verification

重点检查以下尺寸：
- `1280x800`
- `1440x900`
- `1920x1080`

验证项：
- 首屏是否能同时看到主要操作区
- 日志区是否不会把主内容挤出视野
- 底部状态栏是否紧凑且不遮挡内容
- 按钮、输入框和标题是否保持舒适可用

## Risks

- 中屏压缩过度会让界面显得“太紧”
- 状态栏和日志区如果压得过多，信息会显得拥挤
- 单列切换点如果不合适，会在 `1180~1280` 附近出现突然跳变

## Mitigations

- 先把尺寸提炼成变量，再逐步调断点
- 使用少量、明确的断点，不做过多碎片化规则
- 每次调整后都跑 `pnpm test:web` 和 `pnpm build`
- 手动对 `1280x800` 和 `1920x1080` 做重点检查
